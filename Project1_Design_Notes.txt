DAY 6 — MFA VIA CONDITIONAL ACCESS
=========================================

Policy Created:
Name: CA001 — Require MFA for All Users

Configuration:
- Users: All users
- Exclude: GG-BreakGlass
- Cloud apps: All cloud apps
- Grant: Require MFA
- State: Enabled (after pilot)

Rollout Strategy:
1. Disabled Security Defaults (prerequisite)
2. Enabled MFA Registration policy (P2 feature)
3. Created CA001 in Report-only mode
4. Validated with What If tool (3 scenarios)
5. Piloted with GG-IT-Users (4 users)
6. Verified pilot success in sign-in logs
7. Expanded to All Users

What If Tests Performed:
- Regular user (Rahul) → MFA required ✅
- Break-glass (BG1) → No policies applied ✅
- External country (Russia) → MFA required ✅

Verification Results:
- 11 users prompted for MFA on next sign-in ✅
- Break-glass accounts unaffected ✅
- IT team MFA registered ✅

Why Report-only first:
Microsoft's recommended pattern for any CA policy.
Zero risk of accidental lockout.
Real data before enforcement.

Why pilot before broad rollout:
Canary deployment principle.
If something breaks, only 4 users affected, not 16.

Lessons Learned:
- Security Defaults must be OFF for CA to work
- MFA registration policy should come BEFORE enforcement
- What If tool is essential for pre-flight validation
- Break-glass exclusion must be tested explicitly

   Definition: IP range (placeholder)
   Marked as trusted: Yes
   Use case: Future "skip MFA from office" pattern

CA Policies Created:

CA220 — Allow Sign-ins from India Only
- Users: All users
- Exclude: GG-BreakGlass, GG-Travel-Exception-Temporary
- Cloud apps: All
- Condition: Locations → Any location, exclude Trusted-India
- Grant: Block access
- State: On (after report-only validation)

CA221 — Block Sign-ins from Risky Countries
- Users: All users
- Exclude: GG-BreakGlass
- Cloud apps: All
- Condition: Locations → Blocked-RiskyCountries
- Grant: Block access
- State: On (after report-only validation)

Why two policies (layered defense):
- CA220 = allowlist (India only)
- CA221 = denylist (specific risky countries)
- Even if CA220 has exceptions, CA221 still blocks high-risk countries

Travel Exception Process:
- Group: GG-Travel-Exception-Temporary
- Process: Manager approval → Security team adds user → Time-bound
- Automation: Power Automate flow removes expired members daily

What If Tests Performed:
- Indian user from India: ✅ Allowed
- Indian user from US (legitimate travel): ❌ Blocked (correct — needs exception)
- Break-glass from Russia: ✅ Allowed (correct — exclusion works)
- Regular user from North Korea: ❌ Blocked by both CA220 + CA221

Security Posture After Today:
- MFA enforced (CA001) — blocks 99.9% of attacks
- India-only access (CA220) — blocks geographic mismatch
- Risky country denylist (CA221) — defense-in-depth
- Layered approach = defense at multiple levels

Business Value:
- Prevents credential stuffing attacks from outside India
- Aligns with data residency/compliance requirements
- Auditable trail for security questionnaires
- Real-world relevant for Wipfli-style India operations



=========================================
DAY 8 — POLICY INTERACTION ANALYSIS
=========================================

Question raised:
"What if a travel-exception user signs in from North Korea?"

Analysis:

CA220 (India-only):
- Travel exception group excluded → CA220 skipped
- Result: would allow

CA221 (Risky country block):
- Travel exception group NOT excluded → CA221 applies
- Country = North Korea (in risky list)
- Result: BLOCK ✅

Final outcome: BLOCKED by CA221.

Why this design is correct:
- Business need: occasional international travel (CA220 flexibility)
- Security need: never allow sanctioned countries (CA221 strict)
- These are different concerns; should be separate policies

Lesson:
Exclusions are per-policy, not global. This gives surgical control.

Three-tier trust model in our tenant:
- Tier 1 (Break-glass): No location restriction
- Tier 2 (Travel exception): Most countries OK, risky countries blocked
- Tier 3 (Regular users): India only

Extreme edge case:
If someone genuinely needs access from a sanctioned country:
- Create GG-RiskyCountry-Exception-Approved (CISO approval required)
- Add to CA221 exclusion
- Create CA222 with ultra-strict requirements:
  * FIDO2 MFA only (phishing-resistant)
  * Compliant device required
  * Sign-in frequency: 1 hour
  * Continuous Access Evaluation enabled
